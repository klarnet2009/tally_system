# Отладка tally_system

Как понять прямо с контроллера, что работает, а что нет — без логического
анализатора.

## Хаб (ESP32-S3)

### Serial-логи — два порта

Плата ESP32-S3: UART0 — это GPIO 43/44, нативный USB — GPIO 19/20.
Пины E28 (включая GPIO 1/3) сериалу **не мешают**. Логи дублируются на оба
порта — подключайтесь любым кабелем, 115200:

- **Нативный USB** (`Serial`, USB-Serial-JTAG)
- **USB-UART мост девкита** (`Serial0`)

Что пишется:
- Баннер с конфигом: `netId / частота / преамбула / refresh`.
- Каждая попытка инициализации E28 **с причиной отказа** (см. таблицу ниже).
- Переходы ATEM: `connecting... / connected / connect timeout / connection lost`.
- Wi-Fi: connected + IP / FAILED.
- Восстановление радио: `[E28] recovered` / `recovery failed: <причина>`.
- Раз в 10 с строка статуса:
  `[STATUS] up=123s radio=OK(0x43) tx=456 drop=0 q=0 wifi=192.168.1.5 atem=RUNNING prog=0x0001 prev=0x0002`

### Serial-команды (любой из портов)

| Команда  | Действие |
|----------|----------|
| `status` | Немедленно напечатать строку статуса |
| `ping`   | Локатор: 3 ping-пакета камере 1 (то же, что кнопка BOOT) |
| `reinit` | Принудительный пере-init радио с причиной результата |
| `help`   | Список команд |

### OLED

- **ATEM подключён**: статус-бар (Wi-Fi/LoRa/ATEM иконки, IP) + сетка tally.
- **ATEM не подключён**: debug-экран — статус-байт SX1280 с расшифровкой
  режима (STB/RX/TX...), уровни пинов BUSY/DIO1, RSSI/SNR, счётчики TX/Drop,
  диагноз SPI, аптайм.
- **Отказ init при загрузке**: `LoRa FAILED` + причина.

## Причины отказа init радио (E28Radio::initErrorStr)

| Сообщение | Что значит | Что проверять |
|-----------|------------|---------------|
| `BUSY stuck (power?)` | BUSY не опустился за 100 мс | Питание модуля, провод BUSY, землю |
| `MISO low (no 3V3?)` | Статус читается как 0x00 | 3V3 на модуле, короткое на MISO |
| `MISO high (no module?)` | Статус читается как 0xFF | Модуль вообще подключён? MISO/NSS провода |

### Тёплая перезагрузка («модуль как будто не подключён»)

У хаба нет линии аппаратного сброса E28 (`E28_PIN_RESET = -1`). При прошивке
или нажатии EN перезапускается только ESP32 — SX1280 остаётся в прежнем
состоянии (sleep/duty-cycle/TX), и SPI-проба раньше проваливалась. Драйвер
теперь сам будит чип (NSS-импульс) и шлёт слепой `SetStandby` перед повторной
пробой; setup делает 5 попыток, в loop восстановление повторяется каждые 10 с.
Если отказ повторяется и после этого — смотрите причину в логе, это уже железо.

Также радио теперь инициализируется **до** Wi-Fi: TX-всплески Wi-Fi на общей
3V3-шине во время старта 27dBm-модуля — известный источник нестабильного init.

## Слейвы (ESP32-C3, USB-CDC `Serial`, 115200)

Логи: баннер с Camera ID/NetID, init с причиной отказа, `[TALLY] State:N RSSI:-xx`
на смену состояния, `[LINK] Signal lost/restored`, `[LoRa] Recovering.../Recovered/
Recovery failed: <причина>`, у v2 — `[STATUS]` раз в 10 с со счётчиками RX/Fail.

Команды v2: `test` (локатор), `red`/`green`/`off` (подменить состояние),
`beep`, `help`.

Команды v2: `test` (локатор), `red`/`green`/`off` (подменить состояние),
`beep`, `id <1-16>` (задать camera ID в NVS + перезагрузка), `help`.

### Camera ID (runtime, без пересборки)

ID хранится в NVS — **один бинарник на все камеры**. Задать в поле:
- **Кнопка BOOT (GPIO9)**: держать при включении → режим set-ID; короткие
  нажатия отсчитывают 1..8 (LED повторяет счёт вспышками + чирп баззера),
  3 с без нажатий → сохранение.
- **Serial**: `id 5` → сохранить ID=5 и перезагрузиться.

### RX-режим (флаг сборки)

- `tally_slave_v2` — **непрерывный приём** (по умолчанию, приоритет надёжности).
- `tally_slave_v2_lowpower` (`-DPOWER_SAVE`) — duty-cycle (~×2 батарея, но окно
  глухоты при перевзводе). Boot-лог печатает выбранный режим.
- `-DTALLY_QUIET` — полностью отключить баззер (тихие площадки).

### Индикация без сериала

| Сигнал | Значение |
|--------|----------|
| 3 зелёных вспышки + бип (v2) / 3 вспышки (v1) при старте | init OK |
| Init провален → НЕ вечное мигание: слейв уходит в `loop()`, `tryRadioRecover` лечит каждые 10 с | смотри Serial |
| **Красный/зелёный/жёлтый ровно** | tally: PROGRAM / PREVIEW / BOTH (доверяй) |
| **Оранжевый пульс + бип раз в 5 с (v2) / ровное медленное мигание (v1)** | потеря радиосигнала (>3 с тишины) |
| **Белые вспышки поверх цвета (v2) / двойное мигание (v1)** | связь есть, но ATEM завис — цвет устарел, **не доверяй** |
| Синие вспышки + 2400 Гц (v2) / частое мигание (v1) | локатор (ping с хаба) |

Баззер **не звучит, пока камера в эфире** (PROGRAM/BOTH) — живой микрофон не
ловит тон. Локатор/потеря сигнала на живой камере = только визуально.

## Протокол v3 и телеметрия

- 9-байтовый кадр, ниббл версии (старый/новый узел fail-closed → показывает
  signal-lost, а не мусор), **CRC-8** вместо XOR, флаг «источник жив».
- Слейвы шлют `CMD_TELEMETRY` (camId, RSSI, флаг батареи) каждые ~2 с. Хаб
  печатает `[TLM] cam N ONLINE/OFFLINE` на переходах и показывает `reach=0x….`
  в `[STATUS]` — какие камеры реально на связи. `0x0005` = камеры 1 и 3 живы.
- При потере ATEM хаб продолжает heartbeat, но со снятым флагом «источник жив»
  → слейвы показывают «не доверяй» (см. выше), а не уверенный устаревший RED.

## Сквозная проверка радиоканала

1. На хабе: `-DLORA_TEST_MODE` в `tally_hub/platformio.ini` → cam1 мигает
   RED/GREEN раз в секунду на слейве с ID 1.
2. Либо прошить `tally_hub_radiotest` + `tally_slave_v2_radiotest` — радуга
   по эфиру; **обе тестовые прошивки работают на боевой частоте/преамбуле**
   (используют общий `tallyApplyRadioProfile`).
3. `ping` с хаба (serial или кнопка BOOT) → синяя вспышка на камере 1.
4. На хабе смотреть `tx=` (растёт) и `drop=` (НЕ растёт) в `[STATUS]`;
   на слейве — `RX:` в `[STATUS]`.
